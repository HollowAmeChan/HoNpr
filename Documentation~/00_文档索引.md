# HoNpr 文档索引

`Documentation~` 只放系统级设计契约，不放派生表格、不放临时 TODO、不放某个实现文件的注释备忘。

本目录文档用于回答五类问题：

1. HoNpr 为什么这样设计。
2. 每一层的事实来源在哪里。
3. Shader 生成链路的边界是什么。
4. lilToon 迁移应该收敛成哪些 HoNpr 类型。
5. Feature、材质语义和跨 pass 导出应该如何显式化。

## 事实来源规则

人工维护的事实来源：

- `ShaderSystem/Contract/HORP_CONTRACT_INDEX.md`
- `ShaderSystem/Includes/INCLUDE_REGISTRY.honprinclude`
- `ShaderSystem/Features/**/*.honprblock`
- `ShaderSystem/Features/**/*.honprparams`
- `ShaderSystem/Features/**/*.honprui`
- `ShaderSystem/Templates/**/*.honprtemplate`
- `ShaderSystem/Presets/**/*.honprpreset`
- `Shaders/ShaderLibrary/**/*.hlsl`
- `Editor/**/*.cs`

派生文件：

- `ShaderSystem/Templates/TEMPLATE_TABLE.md`
- `ShaderSystem/Features/FEATURE_BLOCK_TABLE.md`
- `ShaderSystem/Presets/PRESET_TABLE.md`
- `ShaderSystem/MaterialUi/MATERIAL_UI_TABLE.md`
- `Shaders/Generated/**/*.shader`

派生文件可以用于审查结果，但不能反向成为设计事实来源。修改结构时，先改源声明，再运行生成器。

## 决策顺序

遇到冲突时按下面顺序判定：

1. 上游 HoRP 契约优先于 HoNpr 本地假设。
2. `*.honprpreset` / `*.honprtemplate` / `*.honprblock` 优先于派生表。
3. ShaderLibrary 手写实现优先承载复杂度。
4. Generated shader 只展示最终装配结果。
5. Inspector UI 永远不能决定 shader 结构。
