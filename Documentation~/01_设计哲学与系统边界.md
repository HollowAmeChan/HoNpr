# HoNpr 设计哲学与系统边界

HoNpr 的目标不是复刻 lilToon 的 Inspector、宏开关和 shader 家族，也不是创建一个用户可以任意拼装的 shader 积木系统。

HoNpr 的目标是建立一套语义明确、生成结果可审查、复杂度可下沉的 NPR 材质系统：

- 用 HoRP 契约定义上游渲染语义。
- 用少量用途明确的 preset 表达材质结构。
- 用 FeatureBlock 声明能力来源和语义输入输出。
- 用 ShaderLibrary 承载复杂 HLSL 实现。
- 用 Template 表达 ShaderLab 和 pass 骨架。
- 用 Generator 组装声明，不创造结构事实。
- 用 Generated shader 展示最终装配图。

一句话边界：

```text
结构来自 preset / template / block，复杂实现下沉到 ShaderLibrary，最终 generated shader 只保留可读装配。
```

更具体地说，shader 不是系统中心，而是材质语义系统的后端输出。HoNpr 先建立可分析的材质语义、feature 所有权、capability 和 pass 需求关系，再生成 Unity 能编译的 shader。

## 设计原则

### 1. 语义优先，不以旧 UI 为中心

HoNpr 先回答“这个材质在 HoRP 中生产什么语义、参与哪些 pass、属于什么用途”，再决定参数和显示方式。

禁止把旧 lilToon 的 UI 折叠项、property 名、hidden pass 名或宏名直接变成 HoNpr 的长期 ABI。

### 2. 少量粗粒度 shader 类型

HoNpr 不按每个功能 toggle 生成一个 shader，也不让一个“全功能母体 shader”在材质实例上运行时裁剪结构。

结构差异应收敛为粗粒度类型：

```text
Domain: Character / Hair / Skin / Environment / FX / Utility
Archetype: Toon / PBR / Transparent / SSS / OIT / Debug
Tier: Lite / Standard / Rich / Specialized
Preset: MaterialPreset.Character_Toon_Standard
Shader: HoNpr/Character/Toon_Standard
```

如果一个开关会改变 pass、feature block、assembly、shader ABI 或 HoRP 资源依赖，它就是新的 preset 或新的专用 shader，不是 UI toggle。

### 3. 复杂度允许存在，但必须在正确层

允许复杂：

- ShaderLibrary 内部算法。
- Assembly 内部预编译组合。
- Generator 的声明校验。
- Preset 对 pass、block、capability 的静态描述。

不允许复杂：

- Generated shader 中的大量 `HONPR_HAS_*` 结构宏。
- Inspector UI 中的结构开关。
- C# 生成器按 preset 名称硬编码 pass 和 block。
- FeatureBlock 中直接写 HLSL 主体。

### 4. 生成结果必须可审查

每个 generated shader 都必须能直接回答：

- 来自哪个 preset。
- 使用哪些 template。
- include 哪个稳定 HLSL 入口。
- 生成哪些 pass。
- 每个 pass 绑定哪个 vertex / fragment entry。
- 不能手改，应该改哪个源声明。

审查 generated shader 时不应该需要阅读一个由生成器临时拼出的隐藏 HLSL 大包。

### 5. UI 是参数编辑层，不是结构来源

Material UI 可以显示当前 preset、参数、贴图、只读 pass 摘要和迁移提示。

Material UI 不能：

- 增删 block。
- 决定 pass 是否存在。
- 写 shader keyword。
- 写 include / define / pragma。
- 通过“修复”按钮改变 shader 结构。
- 自动反射 shader property 生成结构 UI。

### 6. keyword 只服务平台和 RP 必需差异

HoNpr 自己的材质功能不通过 `multi_compile`、`shader_feature` 或 material keyword 表达结构存在性。

允许但必须集中审查的 keyword：

- Unity / URP 要求的基础变体。
- instancing、stereo、DOTS、debug 等系统级需求。
- 少量不可避免的 RP 光照入口差异。

禁止用 keyword 表达 Rim、MatCap、Glitter、Outline、OIT、SSS 等 HoNpr feature 是否存在。

## 核心架构原则

### 1. Feature Ownership 优先

一个 feature 必须能作为完整对象被发现，而不是散落在 HLSL、Inspector、property、RenderFeature、keyword、debug view 和 preset 默认值里。

当前阶段不要求立刻改变所有目录形态，但每个 feature 的以下信息必须显式可追踪：

- 实现入口。
- 参数和默认值。
- UI 分组或 UI profile 引用。
- 依赖。
- capability。
- producer / consumer 关系。
- 编译和导出行为。
- debug 或验证锚点。

如果理解一个 feature 必须跨工程全文搜索，说明它还没有完成所有权收敛。

### 2. 少开关，多层级

HoNpr 不追求任意 feature toggle 的自由组合。自由组合会把复杂度转移成 variant、PSO、缓存、UI 和生成器状态空间。

长期结构应优先落到：

```text
Domain -> Archetype -> Tier -> Preset
```

材质参数控制强度、权重、颜色、贴图和曲线；preset 控制 feature 是否存在、pass 是否存在、资源依赖和 ABI。

### 3. 材质语义优先于固定 AOV

AOV 的本质不是固定 RenderTexture 槽位，而是材质计算过程中的语义流是否需要被某个 pass 访问。

HoNpr 不应建立“某个语义永远对应某个全局 RT”的长期规则。正确方向是：

```text
Pass declares Need
Material declares Supports / Produces
Resolver maps semantic to transient binding
Runtime binds only for current frame and current pass
```

RenderFeature 不应该理解 Rim、MatCap、AnimeShadow、HairShift 这类材质 feature；它只声明 pass 需要哪些语义，由中间层完成 capability resolve 和 transient export binding。

### 4. 显式关系替代潜规则

任何默认存在的关系都必须写成声明或契约，不能靠命名、RT 槽位、keyword、pass 顺序或 UI 分组暗含。

特别禁止：

- keyword 默认打开某个 feature。
- RT 槽位默认存放某个语义。
- pass 默认消费某个未声明结果。
- RenderFeature 默认知道某类材质内部结构。
- UI 折叠项默认等于 shader 能力。

## 越界判定

| 现象 | 判定 | 应该移动到 |
| --- | --- | --- |
| Generated shader 出现大量 `HONPR_HAS_*` 和 `#if defined(...)` | 结构泄漏 | Preset / Assembly |
| Template 内联完整 fragment 主体 | Template 越界 | ShaderLibrary / Assembly |
| FeatureBlock 直接写 `#include`、`#define`、`#pragma` | 声明层越界 | include registry / generator / assembly |
| Generator 按 `Lite`、`Rich` 名称决定 block | 事实来源错误 | `.honprpreset` |
| UI toggle 改变 pass 或 keyword | UI 越界 | 新 preset 或固定参数 |
| ShaderLibrary 重新定义 HoRP semantic | 上游契约越界 | HoRP contract |
| lilToon property 成为新 ABI | 迁移污染 | LegacyInterop 映射 |

## 底线

HoNpr 可以有复杂 HLSL、复杂迁移映射和复杂生成器校验；但面向用户和审查者的结构必须是少量明确 preset、稳定 template、稳定 block、稳定 generated shader。

系统宁可增加一个语义清晰的 preset，也不要增加一个长期难以验证的自由组合开关。
