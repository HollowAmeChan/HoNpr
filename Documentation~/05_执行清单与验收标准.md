# HoNpr 执行清单与验收标准

本文给实现和评审使用。任何新增 shader 能力、preset、block、template、生成器逻辑或迁移规则，都应按本清单检查。

## 新增 FeatureBlock

必须满足：

- 有稳定 `MaterialBlock.*` ID。
- 声明 domain / category / stage。
- 声明输入和输出。
- 声明 required include alias，不直接写裸 include。
- 声明 capability 或解释为什么不需要。
- 声明 producer / consumer 关系，或解释为什么没有跨 feature 语义关系。
- 声明参数、默认值和 UI profile 引用，或能从同一 feature ID 追踪到它们。
- 声明 legacy source 或明确为 HoNpr 原生。
- 有 debug view、验证锚点或审查方式。
- 不写 `#include`、`#define`、`#pragma`。
- 不决定 pass 是否存在。
- 不读取 UI 状态。
- 不依赖 RT 槽位、keyword 或命名约定暗示语义。

失败处理：

- 如果 block 需要改变 pass，新增或修改 preset。
- 如果 block 需要新 HoRP semantic，先补上游契约。
- 如果 block 只是在描述算法细节，把细节放到 ShaderLibrary。

## 新增 Template

必须满足：

- 只描述 ShaderLab / pass / render state / entry slot。
- pass 条件来自 preset token。
- 不内联完整 lighting、lobe 或 composite 主体。
- 不靠 UI 参数改变结构。
- 所有 pragma 来源可解释。
- source mapping 由 generator prepend，不写死在模板里。

失败处理：

- 模板太大：拆到 ShaderLibrary / Assembly。
- pass 差异太多：拆 preset 或专用 template。

## 新增 Preset

必须满足：

- 有 domain / archetype / tier。
- 明确 templates、blocks、passes、generator。
- 明确 generated shader path。
- 明确 capability 和 phase policy。
- 明确状态：active / prototype / deprecated / planned。
- 能解释为什么不是已有 preset 的参数变化。

状态含义：

- `Active`：默认用户入口，可以在新增材质和迁移目标中直接使用；只表示结构、参数、UI 和语义已经表面通过，不承诺每个光照组分都完成视觉调校。
- `Prototype`：可生成、可用于迁移验证，但不作为默认入口。
- `Planned`：结构声明和 generated shader 可以已经存在，但还没按表面通过标准放行；不能因为文件已生成就视为完成。
- `Deprecated`：旧规划或兼容入口；默认生成器不应生成，除非有显式回归需求。

当前生成器实现按“非 `Deprecated` 且声明了 `generator`”生成 shader，因此评审时不能只看是否存在 generated shader，还必须看 status、参数声明、材质 UI、LegacyInterop 和语义归属。

当前阶段的 preset 放行标准是“表面正确”：

- generated shader 能按声明生成出来。
- Properties 和参数声明能对上。
- 材质 UI 白名单完整且不越界控制结构。
- pass、block、capability 和 semantic 归属看起来正确。
- 没有旧 ABI、大段 HLSL 主体或隐藏中间 HLSL 大包泄漏到 generated shader。

不要求在这一阶段逐一证明每个 lighting / stylized lobe 的最终视觉效果完全正确；那属于后续渲染质量、对照测试和美术调参工作。

禁止：

- 为每个 UI toggle 增加 preset。
- 用 preset 模拟任意 block 拼装。
- 让材质实例改变 preset 结构。

判断：

```text
只改数值 = 参数。
改 block / pass / resource / ABI / assembly = preset。
改 HoRP 生命周期 = 上游 contract。
```

## 新增 ShaderLibrary 实现

必须满足：

- include 路径进入 registry 或由 assembly 管理。
- 不重新定义 HoRP 语义。
- 不长期依赖旧 lilToon include。
- 不使用旧 property 作为新 ABI。
- 内部宏不泄漏到 generated shader 作为结构开关。
- 公共 struct / function 命名稳定。

建议：

- lobe 输出使用结构化类型。
- composite 顺序在 assembly 固化。
- 平台差异集中在少数 helper。

## 新增 Assembly

必须满足：

- 位于 `Shaders/ShaderLibrary/Assemblies/`。
- 对每个 preset 输出稳定 entry。
- include 底层 ShaderLibrary。
- 固化 lobe 顺序和 pass entry。
- 内部可以使用宏，但不要求 generated shader 再拼结构。
- 名称包含 domain / archetype / tier。

失败处理：

- Assembly 太宽：拆成 Lite / Standard / Rich / Transparent 等具名 assembly。
- Assembly 仍需 generated shader 大量 `HONPR_HAS_*`：继续下沉结构。

## 修改 Generator

必须满足：

- 只读取声明源和 HoRP contract index。
- 不读取材质 Inspector 当前状态决定结构。
- 不按 preset 名称硬编码 block / pass。
- 不生成 hidden HLSL 大包。
- 写文件统一 UTF-8 和 LF。
- 生成前校验引用。
- 生成后更新派生表。
- generated shader 包含 source mapping。

允许：

- 按 `generator CharacterToonTemplate` 选择组装器。
- 根据 preset 声明选择 template、pass、entry、include。
- 根据 block 声明聚合 capability。

禁止：

- `if presetName.Contains("Rich") then add Glitter`。
- `if material.GetFloat(...) then add pass`。
- 在 C# 中拼完整 fragment 主体。

## 修改 Material UI

必须满足：

- UI profile 来自 `.honprui`。
- 只编辑白名单参数。
- 可以显示只读结构摘要。
- 可以显示迁移提示。
- 不能写 keyword 表示结构。
- 不能增删 block / pass。
- 不能暴露自由 render state。

## 新增或修改材质语义 / Export

必须满足：

- semantic 名称能追溯到 HoRP contract 或 HoNpr 本地声明边界。
- 参数能追踪到 semantic，而不只是 shader uniform 名。
- material 只声明 `Supports` / `Produces`，不保存 runtime export slot。
- pass 只声明 `Need`，不理解材质 feature 内部实现。
- resolver 负责把 `Need` 和 `Supports` 映射到当前 frame / pass 的 transient binding。
- AOV / export 不以固定全局 RT 表作为长期 ABI。
- feature 中间结果进入 Material Context 或等价语义结构，不散落成隐式全局状态。
- RenderGraph 只承载显式数据流和生命周期，不反向拥有材质语义。
- generated shader 不通过全局纹理副作用建立未声明跨 pass 数据链路。

失败处理：

- 如果 semantic 需要 HoRP 生命周期或资源，先补 HoRP contract。
- 如果只是当前 pass 的中间结果，使用 frame-local resolve，不新增全局状态。
- 如果 RenderFeature 需要知道 Rim、MatCap、AnimeShadow 等 feature 名，改成 pass semantic request。

## Generated Shader 验收

必须满足：

- 有 source mapping 头。
- shader name 是 HoNpr 用户语义。
- pass 列表来自 preset。
- include 稳定 ShaderLibrary 或 Assembly。
- entry 名稳定。
- 没有完整 HLSL 主体。
- 没有大量 `HONPR_HAS_*`。
- 没有大量 `#if defined(HONPR_HAS_...)`。
- 没有旧 `_lil*`、`_HoAov*`、`HoAOV` 长期 ABI。
- 不生成 `Shaders/Generated/*.hlsl` 中间文件。

可接受的过渡：

- 迁移 prototype 可暂时暴露更多来源结构，但必须标记 prototype。
- 过渡大模板必须有下沉计划，不能被新 preset 当作长期目标。
- 当前 `CharacterToon` generated shader 允许保留少量由 block 声明聚合出的 `HONPR_HAS_*` define，用于驱动 shared assembly；但 define 数量增加、向新 shader 家族扩散，或在 generated shader 中出现大量 `#if defined(HONPR_HAS_...)`，都应视为需要继续下沉到具名 Assembly。
- generated shader 中的 `HONPR_HAS_*` 只能是迁移期结构胶水，不能成为材质实例 ABI、UI 开关或长期变体策略。

当前验收只判断 generated shader 的结构是否可审查，不判断每个 lobe 的美术表现是否最终正确。

## lilToon 迁移验收

必须满足：

- 旧 shader 能映射到明确 HoNpr preset。
- 旧 property 有 LegacyInterop 记录。
- lilToon 来源型用户 shader 名显式带 `LilToon` 来源身份。
- 来源 block 可以带 `LilToon` 后缀。
- Refraction / Fur / Gem / FakeShadow 不混入 Character Toon 默认系列。
- 迁移原型不作为默认材质入口。

## Review 快速判定

看到以下现象应直接要求修改：

- `Generated/` 下新增 `.hlsl`。
- `Generated/*.shader` 出现大段手写算法。
- `Generated/*.shader` 出现几十个结构宏。
- `.honprblock` 里出现裸 HLSL 预处理行。
- C# 根据 preset 名称决定结构。
- UI 写 keyword 或 pass。
- 新文档把派生表当事实来源。
- 新能力绕过 HoRP contract 直接读写资源。
- feature 的实现、参数、UI、默认值、依赖和 capability 只能靠全文搜索拼出来。
- semantic 通过固定 RT 槽位、keyword、pass 顺序或 UI 折叠项暗示。
- RenderFeature 直接理解材质 feature，而不是声明 pass 需要的 semantic。
- 全局 AOV 状态表替代 frame-local semantic resolve。

## 最终底线

每个能力必须能落到一个明确位置：

```text
语义归 HoRP。
声明归 ShaderSystem。
实现归 ShaderLibrary。
组合归 Preset / Assembly。
装配归 Generator。
跨 pass 导出归 Semantic Resolve。
审查归 Generated shader。
参数归 Material UI。
迁移归 LegacyInterop。
```

放不进这张表的设计，先不要实现。
